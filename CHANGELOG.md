## [Unreleased]

- **Discovery-web Run dropdown:** Run strategy dropdown now populated from GET /v1/strategies (all strategies + All) instead of hardcoded Mock Trending. (discovery-web)
- **AliExpress discovery – Aliexpress True API:** AliExpress strategy now uses Aliexpress True API v2.1.0 (RapidAPI) **GET /hot-products** for trending, high-commission products. Default host `aliexpress-true-api.p.rapidapi.com`; configurable `DISCOVERY_ALIEXPRESS_SHIP_COUNTRY` (AE) and `DISCOVERY_ALIEXPRESS_LANGUAGE` (EN). Response parsed from `products.product`; optional `max_sale_price` from DISCOVERY_MAX_ASP_AED. (discovery-api)
- **Discovery Phase 2 – Amazon, Reddit, TokInsight:** Amazon best-seller by category strategy (`amazon_bestseller_sellermagnet`) via SellerMagnet API; Reddit and TokInsight as optional query sources. Shared `get_queries()` in `query_source.py` dispatches on `DISCOVERY_QUERY_SOURCE` (static, trending, reddit, tokinsight). SerpApi and AliExpress strategies use `get_queries()`. New env: SELLERMAGNET_API_KEY, AMAZON_*, REDDIT_*, TOKINSIGHT/RAPIDAPI_TOKINSIGHT_HOST. README documents Phase 2 env and query source options. (discovery-api)
- **Discovery Phase 1 – static query source:** Shared query source (`app/services/query_source.py`) with category-based default list; DISCOVERY_QUERY_LIST supports comma-separated or file path (one query per line; # comments ignored). Example file `data/queries_example.txt`. Script `scripts/bootstrap_query_list_from_csv.py` to build query list from CSV (e.g. Kaggle). Default remains static; Google Trends for manual research via GET /v1/trends/*. README updated with Phase 1 and bootstrap docs. (discovery-api)
- **Real discovery strategies (Middle East):** SerpApi Google Shopping (UAE) and AliExpress via RapidAPI as pluggable strategies; configurable DISCOVERY_MAX_ASP_AED, DISCOVERY_QUERY_LIST, DISCOVERY_QUERY_SOURCE (static vs trending). Google Trends: GET /v1/trends/trending-searches and GET /v1/trends/interest-by-region (SerpApi). When QUERY_SOURCE=trending, Shopping strategy uses trending searches as queries. (discovery-api)
- **Product discovery system:** New discovery-api (FastAPI, port 8004/8006) and discovery-web (Vite + React, port 5175/5176). Discovery-api uses its own SQLite DB; pluggable strategies (mock strategy included), GET /v1/products (list/filter), GET /v1/strategies, POST /v1/runs. Discovery-web: product list with filters (strategy, delivers to UAE), link to source, trigger run. Docker and docs (ARCHITECTURE, README, docs/DOCKER, PRODUCTION) updated; discovery-api/README and discovery-web/README added. (discovery-api, discovery-web, docker, docs)
- **API Gateway/BFF and structured logging & tracing:** Customer BFF (customer-bff/): FastAPI service that proxies /v1/* to customer-api with X-Request-ID and auth headers, per-IP rate limiting (RATE_LIMIT_PER_MIN), and GET /v1/checkout-context aggregation (cart + addresses). Docker: customer-bff on port 8012; ARCHITECTURE and docs/DOCKER updated. Structured logging: request_id bound to structlog context in customer-api and admin-api; OpenTelemetry (optional) in both APIs with trace_id/span_id in request logs and OTLP export when OTEL_EXPORTER_OTLP_ENDPOINT is set. (customer-bff, customer-api, admin-api, ARCHITECTURE, docs)
- **Production guide (PRODUCTION.md):** Path B (cloud free tiers) expanded to granular step-by-step: Render (PostgreSQL, customer-api, admin-api, seed), Vercel (customer-web, admin-web), CORS, EAS (configure, build, submit). Path C (GCP) added: project and APIs, Cloud SQL, Secret Manager, build/push images, Cloud Run for APIs and frontends, seed, CORS, optional domains and bulk worker. Deployment options table updated with Option C and EAS. (docs)
- **Production guide (PRODUCTION.md):** Rewritten with two paths: (A) Docker Compose on a host (recommended) with PostgreSQL and reverse proxy; (B) Cloud free tiers (Vercel + Render). Checklist and steps updated for current stack (customer-api, admin-api, customer-web, admin-web, bulk-import-worker, init, shop/EAS). References ARCHITECTURE.md and docs/DOCKER.md. (docs)
- **Codebase cleanup:** customer-api no longer exposes parent product `stock_quantity`/`stock_reserved` in product or cart API responses (stock on ProductChild only; `stock_net` and child-level fields kept). CartItemProduct schema: removed `stock_quantity`. Taxonomy `path` comment reworded. customer-web: SavedAddress includes `lat`/`lng`; cart store removed unused `getQuantityForProduct`; ProductCard and PDP use only `stock_net`. shop: same cart-store cleanup and stock_net-only usage. README stock reservation section updated to describe child-level stock. (customer-api, customer-web, shop, README)
- **Image optimization:** customer-web: Next.js `Image` with `fill` and responsive `sizes` for product/order images (ProductCard, PDP, wishlist, cart, checkout, orders); `images.remotePatterns` for Unsplash and backend origins. admin-web: `loading="lazy"` and `decoding="async"` on product and brand images. shop: `expo-image` for product/order images (caching, contentFit). ARCHITECTURE: CDN note for `image_url`. (customer-web, admin-web, shop)
- **Wishlist (parent-product only):** New `WishlistItem` model and `/v1/wishlist` API (GET list, POST add by product_slug, DELETE remove, POST move-to-cart with child_code). Logged-in only; single move-to-cart endpoint adds to cart and removes from wishlist atomically. customer-web: Navbar wishlist link, heart on ProductCard and PDP, `/wishlist` page with grid and move-to-bag (single/multi-size). shop: useWishlistStore, heart on listing and PDP, Wishlist screen (Account menu), move-to-bag. i18n keys in seed and both frontends. (customer-api, customer-web, shop)
- **Docker ports for parallel run:** Docker Compose now publishes APIs and frontends on different host ports (8002, 8003, 3002, 5174) so local dev (8000, 8001, 3000, 5173) can run at the same time without "address already in use." Ports table added to ARCHITECTURE.md; docs/DOCKER.md and README updated. Project rules: when adding a new service, assign the next free port in both local and Docker columns and update ARCHITECTURE, docker-compose, and DOCKER.md. (docker, docs, ARCHITECTURE, .cursor/rules)
- **Shop: same localhost fix as customer-web:** Normalize API base URL at runtime so any `EXPO_PUBLIC_API_URL` containing `localhost` is rewritten to `127.0.0.1`, avoiding IPv6 resolution failures on macOS. README updated to prefer 127.0.0.1. (shop)
- **localhost → 127.0.0.1 audit and run docs:** Admin-web default proxy target and .env.example now use 127.0.0.1:8001; customer-web .env.example and ADMIN_URL default use 127.0.0.1. All docs (ARCHITECTURE, README, docs/DOCKER, app READMEs) use 127.0.0.1 for URLs. Added “Running: local vs Docker” and “macOS / Windows” in ARCHITECTURE; Docker troubleshooting note for localhost/IPv6. (admin-web, customer-web, docs)
- **Backend URL use 127.0.0.1 instead of localhost:** Default API base URL is now `http://127.0.0.1:8000` so connections use IPv4; on many systems `localhost` resolves to IPv6 (::1) while uvicorn binds to 127.0.0.1, causing requests to fail. Updated customer-web (next.config, lib/api), shop (lib/api, .env.example), and READMEs. (customer-web, shop)
- **API error display (customer-web):** All API error handling now uses `getApiErrorDetail()` so validation/error objects are never rendered as React children; login, signup, and profile address forms show a single string message. Backend access: server-side uses `BACKEND_ORIGIN` (Docker: `http://customer-api:8000`) or `http://127.0.0.1:8000/v1`; browser uses rewrites/proxy so frontends work locally and in Docker. (customer-web)
- **Docker Compose design-tokens fix:** customer-web and admin-web images now build from repo root context so the `@ecommerce/design-tokens` workspace package is available. Updated docker-compose (context + dockerfile), both Dockerfiles (copy root package.json, lockfile, packages, workspaces; npm ci; WORKDIR app), added root .dockerignore, and noted in ARCHITECTURE. (docker, customer-web, admin-web)
- **Idempotency, retries, timeouts:** Idempotency keys for order creation and payment intents (customer-api: `idempotency_keys` table, Idempotency-Key header; customer-web and shop send key on checkout). Structured retries with backoff for Stripe (tenacity, customer-api). Request timeouts: DB connect timeout (customer-api, admin-api), Stripe call timeout (customer-api), axios 30s (customer-web, admin-web; shop already had it). (customer-api, admin-api, customer-web, admin-web, shop)
- **Shop pull-to-refresh:** Home, Cart, and Orders screens support pull-from-top refresh on iOS/Android via RefreshControl; full-screen loader only on initial load when list is empty. (shop)
- **Seed: 5 single-sized + 5 multi-sized products:** PRODUCTS extended with optional 11th element `children_spec`; single-sized keep one child with `single_size`; 5 new multi-sized products (Fashion/Outdoors) have multiple ProductChild rows with size_value S/M/L/XL, barcode, and stock. (customer-api)
- **customer-web API 404 fix:** Normalize `NEXT_PUBLIC_API_URL` so full URLs (e.g. `http://localhost:8000`) get `/v1` appended; requests then hit `/v1/i18n`, `/v1/products`, `/v1/cart` correctly. Updated .env.example and README for API URL. (customer-web)
- **CORS OPTIONS 400 fix:** Customer-api returned 400 on CORS preflight (OPTIONS) when the request Origin was not in the allowed list. Added `allow_origin_regex` in development so any `http(s)://localhost` or `127.0.0.1` (any port) is allowed; added 5173 to default origins; validator to parse `BACKEND_CORS_ORIGINS` from comma-separated env. (customer-api)

## 2025-02-09

- **Rules:** Project standards updated: README files must stay up to date with launch instructions; new apps require an app-specific README and main README update; all changes must be logged in CHANGELOG.md only (no duplicate changelog elsewhere). CHANGELOG.md restructured to consistent format (date sections, `- **Topic:** Description. (scope).`). (.cursor/rules, CHANGELOG.md)
- **Docs:** Changelog moved from README to CHANGELOG.md. README and app READMEs updated with accurate run steps (Customer API first for customer-web/shop; Admin API first for admin-web), API base URLs (`/v1`), and setup notes (ensure_tables, ensure_admin, reset_and_seed). (docs, customer-api, admin-api, customer-web, admin-web, shop)
- **Product parent–child (variants/sizes):** Every product has at least one child (ProductChild). Parent has code, content, price; children have code, barcode, size_value ("single_size" or S/M/L), stock. Single-sized products: one child with size_value "single_size" (user provides barcode only in admin/bulk). Cart, order, and stock reservation always reference product_child_id (required). (customer-api, admin-api, admin-web, customer-web, shop)
- **Customer API:** PDP returns children and single_sized; product list aggregates stock from children; cart and order APIs require child_code; reservations on child. (customer-api)
- **Admin API:** Product create with code generation and at least one child (single-size or multi-size); children CRUD; order status updates child stock on shipped/cancelled. (admin-api)
- **Admin-web:** Product form "Sizes" section with single-size option (barcode only) or multiple sizes; list/add/edit/delete children; product table shows code and aggregated stock. (admin-web)
- **Bulk import:** Option A only—same file with optional columns parent_code, size_value, barcode, is_single_size; single-size rows create one child with size_value "single_size"; multi-size rows create parent+child or add child. (admin-api)
- **Seed:** New structure only; every product has code and one child (single_size); order items and seed data use product_child_id. (customer-api)
- **customer-web & shop:** PDP size selector when multiple sizes; add to cart and cart/checkout use child_code; ProductCard uses first child when one variant. (customer-web, shop)
- **Listing and PDP add-to-cart:** Product list shows only products with at least one child in stock. Listing: single-sized products show Add to cart (and qty); multi-sized show "View variants" linking to PDP. PDP: single-sized show Add to cart only; multi-sized show "Select size" (scrolls to sizes) until a size is selected, then "Add to cart"; out-of-stock sizes disabled. (customer-api, customer-web, shop)
- **Loading and transition consistency:** Unified skeleton vs spinner rules; PageLoader, ProductCardSkeleton, PdpSkeleton (customer-web); TableSkeleton (admin-web); FullScreenLoader (shop). Transitions aligned with design_template. (customer-web, admin-web, shop)
- **Design tokens single source:** `packages/design-tokens` with tokens.json, generated theme.css, and JS export. customer-web and admin-web import theme; shop uses package in Colors. Rule: token changes only in packages/design-tokens. (packages/design-tokens, customer-web, admin-web, shop)
- **Taxonomy & Attributes Admin:** Admin-web can create, update, and deactivate taxonomies (Taxonomies page); create, update, and deactivate taxonomy attributes and options (Attributes page). Added `is_active` to Taxonomy, TaxonomyAttribute, TaxonomyAttributeOption. Customer-facing endpoints filter by `is_active`. Run `reset_and_seed.py` to add new columns to existing DB. (admin-api, admin-web, customer-api)
